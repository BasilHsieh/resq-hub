# ResQ Hub — Claude 操作指引

## 專案背景

Basil 是軟體產品經理，曾三次進入 2025 花蓮光復災區（馬太鞍溪堰塞湖溢流）擔任志工。
本專案目標：建立台灣災害協作的知識庫，並最終發展成數位工具。

Basil 不寫 code，由 Claude 負責技術實作與文件處理。
溝通語言：**繁體中文**。

---

## 知識庫架構

```
raw/              Basil 丟進來的原始資料（不進 git，不動，只新增）
knowledge/
  context/        背景研究（論文整理、國內外案例）
  framework/      核心框架（角色模板、資訊流架構）
  fieldwork/      現場經驗與訪談整理
output/
  vision.md       專案 Vision
  prd/            數位工具規格
archive/          舊資料，不進 git，不動
INDEX.md          知識庫主索引（Claude 負責維護）
CLAUDE.md         本檔
```

`raw/` 和 `archive/` 都在 `.gitignore` 裡，不會上 git。
有價值的內容是 Claude 從 `raw/` 處理後放入 `knowledge/` 的版本，不是原始檔本身。

---

## 處理原始資料的流程

當 Basil 在 `raw/` 放入新文件，或直接在對話中提供資訊：

1. **理解內容**：判斷文件性質（論文、訪談、會議記錄、個人想法等）
2. **有疑問先問**：不確定分類或詮釋時，先跟 Basil 討論，不要自行假設
3. **整理成結構化文件**：放入對應的 `knowledge/` 子目錄
4. **加上 frontmatter**：每份知識文件需有 tags 和 summary
5. **更新 INDEX.md**：加入新文件的摘要與 tag

### 文件 frontmatter 格式

```markdown
---
tags: [tag1, tag2, tag3]
source: raw/原始檔名.md   （若有原始檔）
date: YYYY-MM-DD
summary: 一句話說明這份文件是什麼
---
```

## Tag 系統

Tags 分兩種：**主題 tag** 和 **類型 tag**

### 類型 tag（文件是什麼）
- `#經驗` — 現場第一手紀錄或訪談
- `#研究` — 論文、報告、國內外案例
- `#框架` — 可被複製的模型、原則、架構
- `#工具規格` — 數位工具的需求與規格
- `#vision` — 專案方向與定位

### 主題 tag（文件在談什麼）
- `#資訊流` — 資訊如何流動、斷點在哪
- `#志工協調` — 志工角色、調度、媒合
- `#組織間協作` — NGO 之間、NGO 與政府之間
- `#散兵志工` — 無組織歸屬的個人志工
- `#物資管理` — 物資調度與追蹤
- `#花蓮光復` — 本次救災的具體案例
- `#台灣脈絡` — 台灣特有的制度、文化、現實
- `#國際比較` — 美日等國案例與比較

> Tag 清單會隨知識庫成長而新增，更新時同步修改本檔。

## INDEX.md 維護規則

每次新增或修改知識文件後，更新 INDEX.md 的對應區塊：
- **文件索引**：加入新檔，附一行摘要
- **Tag 索引**：在對應 tag 下加入檔案連結

INDEX.md 要保持「讀一個檔案就能掌握整個知識庫狀態」的程度。

---

## Token 管理原則

- 大型原始檔（>1000 行）不全讀，先讀前 80 行判斷結構，再決定讀哪段
- 每次 session 開始：讀 CLAUDE.md + INDEX.md，其他檔案按需讀取
- 知識文件的 summary 要夠好，讓我不需要重讀全文就能判斷相關性
